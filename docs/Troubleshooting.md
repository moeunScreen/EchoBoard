
## 프로젝트 구조 파악 및 문서화의 어려움 

### 문제
프로젝트를 다시 정리하면서 API와 DB 구조를 문서화해야 했습니다
하지만 프로젝트를 개발한 지 시간이 지나면서 
전체 흐름을 한 번에 파악하기 어려웠습니다
### 원인
초기 개발 당시에는 README와 API 명세, ERD를 별도로 작성하지 않아 
코드를 직접 따라가며 구조를 분석해야 했습니다
### 해결
Controller -> Service -> Repository 순으로 요청 흐름을 추적
### 결과
프로젝트 구조를 빠르게 이해할수있는 문서를 구축했고,
포트폴리오 활용성이 향상됨

## Thymeleaf 페이지 요청과 Data Request가 혼재

### 문제
프로젝트를 다시 문서화하는 과정에서 Web Request를 정리하려고 하였으나,
Thymeleaf 화면을 반환하는 요청과 
Ajax를 통해 데이터를 반환하는 요청이
같은 Controller에 혼재되어 있어 요청의 성격을 한눈에 파악하기 어려웠습니다

### 원인 
초기에는 기능 구현에 집중하여 Controller를 작성하였기 때문에
요청 유형을 구분하는 기준이나 별도의 API 문서가 존재하지 않았습니다
이로 인해 Controller만 확인해서는
- 화면 이동을 위한 요청인지
- JSON 데이터를 반환하는 요청인지
즉시 구분하기 어려웠습니다

### 해결 과정
문서화 과정에서 모든 요청을 분석하여 다음과 같이 분리하였습니다

- **Page Request**
    - Thymeleaf View 반환
    - 화면 이동을 담당하는 요청

- **Data Request**
    - JSON 또는 객체 반환
    - Ajax를 통한 비동기 데이터 요청
### 결과
Page Request와 Data Request를 분리하여 API 명세 작성
요청 유형을 분류하여 Web Request 문서화
이를 통해 Controller 요청 흐름과 Endpoint의 역할을 쉽게 
파악할 수 있도록 개선하였습니다

## DB 초기화 이후 Redis 캐시 데이터 불일치 문제

### 문제
개발 과정에서 데이터베이스를 초기화한 이후,
조회수 데이터가 예상과 다르게 표시되는 문제가 발생하였습니다

### 원인
조회수 기능은 Redis를 캐시 저장소로 사용하고 있었는데,
DB 데이터는 초기화되었지만 Redis에 기존 캐시 데이터가 남아있었습니다
DB조회수와 Redis조회수가
값이 서로 다른 상태가 발생하였습니다

### 해결 과정
기존 Redis에 저장된 조회수 Key를 삭제하여 이전 캐시 데이터를 제거하였습니다
이후 게시글 조회 과정에서 DB 데이터를 기준으로
Redis 캐시가 다시 생성되도록 하였습니다
- DB 조회한 값을 기준으로 Redis Key 재생성
- DB와 Redis의 조회수 값이 일치하는지 확인

### 결과
Redis와 DB의 조회수 값을 비교하여 데이터가 정상적으로 
일치하는 것을 확인하였습니다.
캐시는 원본 데이터(DB)와 별도로 관리되기 때문에,
원본 데이터 변경 시 캐시 무효화 또는 재동기화 과정이 
필요하다는걸 확인했습니다
